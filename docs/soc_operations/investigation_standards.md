# Investigation Standards

Every CDET investigation must meet the standards in this document before a ticket may be closed. These requirements apply to all True Positive and Needs Investigation dispositions. False Positives require only the quality gate fields in Section 6.

---

## 1. Required Evidence for Every Investigation

Collect the following for every investigation regardless of detection type. Missing items must be explicitly explained in the ticket (e.g., "Source IP not available — event generated by AWS service principal").

| Evidence Item | Splunk Field | Required |
|--------------|-------------|----------|
| Detection ID | Alert metadata (`rule_name`, CDET ID format: `CDET-YYYY-NNNN`) | Yes |
| Actor ARN | `userIdentity.arn` | Yes |
| Actor type | `userIdentity.type` (IAMUser, AssumedRole, Root, AWSService, etc.) | Yes |
| Access key ID | `userIdentity.accessKeyId` | If programmatic access |
| Session token ARN | `userIdentity.sessionContext.sessionIssuer.arn` | If assumed role |
| Source IP address(es) | `sourceIPAddress` | Yes (note if AWS internal) |
| User agent string | `userAgent` | Yes |
| Event time (UTC) | `eventTime` | Yes |
| AWS region | `awsRegion` | Yes |
| Affected resource ARN | `requestParameters` (varies by API) | Yes |
| Event name(s) | `eventName` | Yes |
| Error code (if any) | `errorCode` | If present |
| Request parameters | `requestParameters` (raw JSON) | Yes |
| Response elements | `responseElements` (raw JSON) | Yes |
| Full event timeline | All events for actor, prior 24 hours | Yes |

## 2. Chain of Custody Requirements for Forensic Evidence

Forensic evidence collected during an investigation must be handled according to the following chain of custody requirements to preserve admissibility and integrity.

1. All evidence files must be named using the convention: `<CDET-ID>_<evidence-type>_<YYYYMMDD-HHMM-UTC>.<ext>`
   Example: `CDET-2026-0042_cloudtrail_timeline_20260618-1430-UTC.csv`

2. Calculate a SHA-256 hash of each evidence file at the time of collection:
   ```
   sha256sum <filename>
   ```
   Record the hash in the incident ticket Evidence Log table.

3. Upload all evidence files to the designated secure evidence bucket (see SOC Lead for bucket name and access policy). Do not store forensic evidence only on a local workstation.

4. The Evidence Log table in the incident ticket must contain one row per artifact:

   | Artifact Name | Type | Collected By | Collected At (UTC) | SHA-256 | Storage Location |
   |--------------|------|-------------|-------------------|---------|-----------------|
   | `CDET-2026-0042_cloudtrail_timeline_...csv` | CloudTrail Export | Analyst Name | 2026-06-18T14:30Z | `<hash>` | s3://evidence-bucket/... |

5. No evidence file may be modified after collection. If analysis requires transformation (e.g., filtering a large CSV), work on a copy and note in the ticket that the original is unmodified.

6. Evidence must be retained for a minimum of 90 days for True Positive incidents and 30 days for inconclusive investigations. The SOC Lead may extend retention for active legal or compliance matters.

## 3. Splunk Pivot Queries by Evidence Type

Use these pivot queries as starting points. Adjust time ranges and filters as needed. All queries assume the CloudTrail index is `index=cloudtrail`.

### Actor Activity Timeline
```
index=cloudtrail userIdentity.arn="<actor_arn>"
earliest=-24h latest=now
| table _time, eventName, awsRegion, sourceIPAddress, userAgent, errorCode, requestParameters
| sort _time
```

### Source IP Enrichment — All Actors Using This IP
```
index=cloudtrail sourceIPAddress="<source_ip>"
earliest=-7d latest=now
| stats count by userIdentity.arn, eventName, awsRegion
| sort -count
```

### Access Key Usage History
```
index=cloudtrail userIdentity.accessKeyId="<key_id>"
earliest=-30d latest=now
| table _time, userIdentity.arn, eventName, awsRegion, sourceIPAddress
| sort _time
```

### Assumed Role Session Pivot — All Actions Under a Session
```
index=cloudtrail userIdentity.sessionContext.sessionIssuer.arn="<role_arn>"
  userIdentity.sessionContext.webIdFederationData.federatedUserId="<session_id>"
earliest=-24h latest=now
| table _time, eventName, awsRegion, sourceIPAddress, requestParameters
| sort _time
```

### Affected Resource History — Who Touched This Resource
```
index=cloudtrail requestParameters="*<resource_id>*"
earliest=-30d latest=now
| table _time, userIdentity.arn, eventName, awsRegion, sourceIPAddress
| sort _time
```

### Error Code Analysis — Failed Attempts by Actor
```
index=cloudtrail userIdentity.arn="<actor_arn>" errorCode!=""
earliest=-24h latest=now
| stats count by errorCode, eventName, awsRegion
| sort -count
```

### Geolocation Pivot — Approved Region Comparison
```
index=cloudtrail userIdentity.arn="<actor_arn>"
earliest=-30d latest=now
| stats count by awsRegion
| join awsRegion [| inputlookup approved_regions.csv | rename region as awsRegion | eval approved="yes"]
| fillnull value="no" approved
| table awsRegion, count, approved
| sort -count
```
The `approved_regions.csv` lookup lives at `splunk/lookups/approved_regions.csv`.

### Correlated Alerts — Same Actor, Last 7 Days
```
index=notable actor_arn="<actor_arn>"
earliest=-7d latest=now
| table _time, rule_name, severity, status
| sort _time
```

## 4. Documentation Standards — What Goes in the Incident Ticket

Every incident ticket must contain the following sections. Use the section headings below verbatim so that reporting and metrics tools can parse them consistently.

### Alert Summary
- Detection ID (format: `CDET-YYYY-NNNN`)
- Alert name (from Splunk rule)
- MITRE ATT&CK tactic and technique (e.g., TA0001 - T1078.004)
- Severity
- Alert time (UTC)
- Assigned analyst
- Ticket opened time (UTC)

### Actor Information
- Actor ARN
- Actor type
- Access key ID (if programmatic)
- Session token ARN (if assumed role)
- MFA used: Yes / No / N/A

### Activity Summary
- Plain-language description of what was observed (2-5 sentences)
- Event names observed
- AWS region(s) involved
- Source IP address(es) and any geolocation or threat intel notes
- Affected resources (ARNs)
- Time range of suspicious activity

### 3-Question Triage Results
- Question 1 (Approved lookup): Result and supporting evidence
- Question 2 (MFA): Result and supporting evidence
- Question 3 (Region): Result and supporting evidence

### Timeline
Chronological list of all relevant events, format: `YYYY-MM-DDTHH:MM:SSZ | eventName | awsRegion | sourceIPAddress | notes`

### Evidence Log
Table per Section 2 requirements above.

### Analysis
- What the actor did and the assessed intent
- Whether the activity matches a known attack pattern (reference MITRE technique if applicable)
- Detection playbook reviewed: `docs/detection_engineering/<detection_id>.md`
- Correlated alerts: list any related CDET IDs

### Disposition
- True Positive / False Positive / Inconclusive
- If True Positive: impact assessment, affected accounts/resources

### Containment and Remediation
- Actions taken (with timestamps and approving authority for destructive actions)
- Actions recommended but not yet taken (and reason for deferral)

### Recommendations
- Detection tuning recommendations (if false positive pattern identified)
- Control improvement recommendations
- Follow-up tasks with owner and due date

## 5. When to Involve AWS Support or Engage AWS IR

### Involve AWS Support (General) When:
- The investigation requires CloudTrail data from a region or time period outside your current Splunk ingestion window
- You need to confirm whether an API call was made by an AWS service on behalf of your account
- There is suspected abuse of the AWS account from outside your organization that may require account-level action
- A resource was created or modified and you cannot determine the actor from available logs

To open an AWS Support case, use the AWS Console under Support > Create Case. Include the CDET Detection ID and your case description in the ticket subject line. Notify the SOC Lead before opening a Support case.

### Engage AWS Incident Response (AWS IR) When:
Any of the following conditions are true:

1. You have confirmed a breach of AWS root account credentials.
2. There is evidence of persistent access mechanisms (backdoor IAM users, rogue cross-account roles) that you cannot fully enumerate.
3. You suspect data exfiltration from S3 or RDS at scale.
4. You have reason to believe the adversary has persistence in the AWS account that survives credential rotation.
5. An AWS Support case has been open for more than 4 hours without resolution during an active incident.

To engage AWS IR:
1. Log in to AWS Console and navigate to AWS Security Hub or open an AWS Support case with severity "Business Critical."
2. Request escalation to the AWS Abuse or AWS Incident Response team.
3. Notify the CISO before engaging AWS IR — this is a significant action that may have contractual and disclosure implications.
4. Document the case number and all communications in the incident ticket.

## 6. Quality Gate — Minimum Fields Required Before Closing a Ticket

A ticket may not be set to "Closed" status in Splunk or the ticketing system until all required fields below are populated. The SOC Lead or Senior Analyst performs a quality review before closure for all True Positive tickets.

### Minimum Required Fields for All Ticket Closures

- [ ] Detection ID (`CDET-YYYY-NNNN`) present
- [ ] Disposition set (True Positive / False Positive / Inconclusive)
- [ ] Actor ARN documented
- [ ] Event time (UTC) documented
- [ ] At least one source IP documented (or explicitly noted as unavailable)
- [ ] Affected resource(s) documented (at least one ARN)
- [ ] 3-Question Triage results documented (all three questions answered)
- [ ] Activity Summary section complete (plain-language description present)
- [ ] Timeline section contains at least the triggering event

### Additional Fields Required for True Positive Closures

- [ ] Full 24-hour actor timeline exported and attached (Evidence Log entry present)
- [ ] Containment and Remediation section complete — all actions taken documented with timestamps
- [ ] If containment approval was required: approval message (Slack link or PagerDuty note) referenced
- [ ] Recommendations section complete with at least one follow-up action or explicit statement that none are required
- [ ] SOC Lead or Senior Analyst quality review signature (name + timestamp) in ticket comments

### Additional Fields Required for False Positive Closures

- [ ] False positive rationale documented — which condition in the detection playbook applies, or plain-language explanation
- [ ] If this is the second or subsequent false positive from the same rule within 30 days: detection tuning request filed (ticket number referenced)

Tickets that do not meet the quality gate must be returned to "In Progress" status until all required fields are complete.
