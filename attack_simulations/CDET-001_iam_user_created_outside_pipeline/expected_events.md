# CDET-001 — Expected CloudTrail Events

## Overview

Executing this simulation generates between 2 and 4 CloudTrail events depending on which variant is used. All events originate from the `iam.amazonaws.com` event source and appear in the `us-east-1` region's CloudTrail trail (or the home region trail if multi-region is enabled).

---

## Event 1: CreateUser

| Field | Value |
|-------|-------|
| `eventSource` | `iam.amazonaws.com` |
| `eventName` | `CreateUser` |
| `requestParameters.userName` | `svc-backup-agent-test` (or chosen username) |
| `requestParameters.tags` | `[{Key: CreatedBy, Value: SimulationTest}]` |
| `userIdentity.type` | `IAMUser` or `AssumedRole` |
| `userIdentity.arn` | ARN of the principal executing the simulation |
| `responseElements.user.arn` | `arn:aws:iam::ACCOUNT_ID:user/svc-backup-agent-test` |
| `responseElements.user.userId` | Unique IAM user ID (AIDA...) |
| `errorCode` | *(absent on success)* |

**SPL Detection Mapping:**
- Primary trigger field: `eventName = "CreateUser"`
- Identity correlation field: `userIdentity.arn` — compared against approved pipeline role ARNs
- User created field: `responseElements.user.userName` — used for subsequent event correlation

---

## Event 2: CreateLoginProfile

| Field | Value |
|-------|-------|
| `eventSource` | `iam.amazonaws.com` |
| `eventName` | `CreateLoginProfile` |
| `requestParameters.userName` | `svc-backup-agent-test` |
| `requestParameters.passwordResetRequired` | `false` |
| `userIdentity.arn` | ARN of the attacker principal |

**Note:** The password value is **never** logged in CloudTrail. Only the username and reset flag appear.

**SPL Detection Mapping:**
- Secondary enrichment signal — a user with `CreateLoginProfile` shortly after `CreateUser` from the same `userIdentity.arn` indicates console access being established
- SPL field: `eventName = "CreateLoginProfile"` correlated by `requestParameters.userName`

---

## Event 3A: AttachUserPolicy (Direct Admin Grant)

| Field | Value |
|-------|-------|
| `eventSource` | `iam.amazonaws.com` |
| `eventName` | `AttachUserPolicy` |
| `requestParameters.userName` | `svc-backup-agent-test` |
| `requestParameters.policyArn` | `arn:aws:iam::aws:policy/AdministratorAccess` |
| `userIdentity.arn` | ARN of the attacker principal |

**SPL Detection Mapping:**
- CDET-004 also fires on this event
- Field `requestParameters.policyArn` contains `AdministratorAccess` or `*FullAccess*`
- Combined with Event 1 within a time window = high-confidence persistence + escalation chain

---

## Event 3B: AddUserToGroup (Group-Based Grant)

| Field | Value |
|-------|-------|
| `eventSource` | `iam.amazonaws.com` |
| `eventName` | `AddUserToGroup` |
| `requestParameters.userName` | `svc-backup-agent-test` |
| `requestParameters.groupName` | `Administrators` (or target group name) |
| `userIdentity.arn` | ARN of the attacker principal |

**Note:** This event does NOT reveal the permissions of the group. A separate `GetGroup` or `ListAttachedGroupPolicies` call is needed to determine the effective permissions granted.

**SPL Detection Mapping:**
- Field: `requestParameters.groupName` — enriched against a lookup table of privileged groups
- Lower-fidelity signal alone; high-fidelity when correlated with Event 1

---

## Event 4: CreateAccessKey

| Field | Value |
|-------|-------|
| `eventSource` | `iam.amazonaws.com` |
| `eventName` | `CreateAccessKey` |
| `requestParameters.userName` | `svc-backup-agent-test` |
| `responseElements.accessKey.accessKeyId` | `AKIA...` (key ID, NOT secret) |
| `responseElements.accessKey.status` | `Active` |
| `userIdentity.arn` | ARN of the attacker principal |

**Note:** The `SecretAccessKey` is **never** logged in CloudTrail. Only the `accessKeyId` appears.

**SPL Detection Mapping:**
- CDET-002 detection logic covers this event when it occurs for existing users
- Field: `responseElements.accessKey.accessKeyId` — used to track which key was created for later abuse correlation

## Event Correlation Chain (SPL)

```spl
index=aws_cloudtrail eventSource="iam.amazonaws.com"
  eventName IN ("CreateUser","CreateLoginProfile","AttachUserPolicy","CreateAccessKey","AddUserToGroup")
| eval new_user=coalesce(
    requestParameters.userName,
    responseElements.user.userName
  )
| stats 
    values(eventName) as events,
    values(userIdentity.arn) as actors,
    min(_time) as first_seen,
    max(_time) as last_seen
    by new_user
| where mvcount(events) >= 2
| eval attack_chain=mvjoin(events, " -> ")
| table new_user, attack_chain, actors, first_seen, last_seen
```

This query surfaces users that triggered multiple IAM events in sequence, indicating automated or scripted account creation.

## Cleanup Events

The following events are generated by the cleanup phase and can be used to verify the simulation ran end-to-end:

| `eventName` | Description |
|-------------|-------------|
| `DetachUserPolicy` | Policy removed from user |
| `RemoveUserFromGroup` | User removed from group |
| `DeleteAccessKey` | Access key deleted |
| `DeleteLoginProfile` | Console access removed |
| `DeleteUser` | User account deleted |
